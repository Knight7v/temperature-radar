# 和风天气API接入方案

## 需求总结
- **API类型**: 未来7天天气预报
- **更新频率**: 每6小时自动刷新
- **失败处理**: 显示错误提示
- **覆盖范围**: 196个城市
- **API配额**: 每天1000次（免费版）

## 配额计算
- 每6小时更新 = 每天4次
- 131个城市 × 4次 = **524次调用/天**
- 在1000次配额内，有476次余量用于错误重试等

## 方案对比

### 方案A：直接替换（推荐）
**描述**: 直接替换 `generateCityData()` 和 `generateAllCityData()` 函数，从和风API获取真实数据，其他代码保持不变。

**优势**:
- 改动最小，只需修改数据生成逻辑
- 所有现有功能（ADI计算、等级判断、排行榜等）无需修改
- 代码结构清晰

**劣势**:
- 依赖外部API，网络问题会导致无数据
- API返回的数据结构需要适配到现有数据格式

**实施难度**: 低

### 方案B：双模式切换
**描述**: 添加API模式/模拟模式切换开关，默认使用API数据，API失败时可切换到模拟数据。

**优势**:
- 灵活性高，可随时切换数据源
- 开发调试方便

**劣势**:
- 需要额外的UI控制
- 增加代码复杂度

**实施难度**: 中

### 方案C：缓存混合模式
**描述**: API数据获取后缓存到localStorage，下次打开时优先使用缓存数据，缓存过期后再调用API更新。

**优势**:
- 减少API调用次数
- 提升页面加载速度
- 有离线数据可用

**劣势**:
- 需要处理缓存失效逻辑
- 增加复杂度

**实施难度**: 中

---

## 推荐：方案A（直接替换）

**理由**:
1. 用户需求明确：就是要真实天气数据
2. 配额计算证明可行（784次/天 < 1000次）
3. 改动最小，风险可控
4. 符合YAGNI原则

---

## 数据适配策略

和风API返回的数据需要映射到现有数据结构：

```
和风7天预报 → 现有数据结构
- tempMax (白天最高温) → maxTemp
- tempMin (夜间最低温) → nightMinTemp  
- tempMin (夜间最低温) + 湿度计算 → feelsLike
- fxDate (预报日期) → future3DayMaxTemps, future3NightMinTemps
```

## 实施范围

**需要修改的文件**:
1. `js/core.js` - 添加API调用函数，修改数据生成逻辑
2. `js/app.js` - 添加定时刷新逻辑
3. `index-v2.html` - 添加数据加载状态提示
4. `css/styles-v2.css` - 添加加载状态和错误提示样式

**保持不变**:
- 所有ADI计算逻辑
- 所有等级判断逻辑
- 所有UI渲染逻辑
